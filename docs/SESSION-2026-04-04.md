### 2. Wykryte bugi scoringu

**Bug #1**: LLM mięknie przy sympatycznych kandydatach

Test: 10 symulacji, Claude Haiku.
Wynik: 4/10 decyzji niespójnych ze skalą. Przykłady:
- Kasia (40 pkt) → "Rozmowa" zamiast "Zadanie tech"
- Bartek (32 pkt) → "Zadanie tech" zamiast "Do przemyślenia"
- Natalia (22 pkt) → "Do przemyślenia" zamiast "Odrzucony"

**Fix**: usunięto `decyzja` i `wynik_lacznie` z tool schema. Serwer liczy deterministycznie.

**Rezultat**: 0/8 niespójności w re-teście.

**Bug #2**: CAP-y w promptach były tylko opisowe

Test: Bartek Kowalski (bez portfolio + bez buildera) dostał 12/25 tech.
Powinno być MAX 8 wg zasady "brak portfolio".

**Fix**: przeformatowano dyskwalifikujące jako CAP-y:
```
Brak portfolio → wynik_techniczny MAX 8 (niezależnie od reszty)
```

**Rezultat**: Bartek po fixie → 8/25 tech.

**Bug #3**: GPT-4o-mini flaky scoring

Test: Michał Stępień (senior z 40 workflow n8n) → 0/55 Odrzucony.
Notatki opisywały go bardzo pozytywnie.

**Fix**: safety check w `route.ts`:
```ts
if (wynik_techniczny === 0 && wynik_komunikacja === 0 && turns > 5) {
  return { success: false, error: 'Oceń ponownie...' };
}
```

**Rezultat**: w drugim runie Michał dostał 49/55 Rozmowa.

**Bug #4**: GPT-5-mini nieużywalny dla chatów

Test: pierwsza symulacja na GPT-5-mini → pętla 50 tur, $0.18.

**Przyczyna**: reasoning tokens zużywają całe `max_completion_tokens`, model zwraca puste odpowiedzi.

**Decyzja**: nie używamy GPT-5-mini. Zostajemy przy gpt-4o-mini.
